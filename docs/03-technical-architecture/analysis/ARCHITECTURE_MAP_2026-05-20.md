## 1. Server Topology

Two FastAPI applications exist. Both default to port 8000 — they are **mutually exclusive** (cannot run simultaneously).

```
┌─────────────────────────────────────────────────┐
│  App A: apps/backend/main.py (516 lines)        │
│  ──────────────────────────────────────────────  │
│  Routes: 8 HTTP + 1 WS                          │
│  Middleware: CORSMiddleware,                     │
│             EncryptedCommunicationMiddleware     │
│  Port: 8000 (default, --port CLI arg)           │
│  Includes: api/router.py (shared router)        │
│  Role: System management + mobile endpoint      │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│  App B: apps/backend/src/services/              │
│         main_api_server.py (1636 lines)         │
│  ──────────────────────────────────────────────  │
│  Routes: 14 direct + all shared router routes   │
│          + state_matrix (34 routes)             │
│          + atlassian (6 routes)                 │
│  Middleware: CORSMiddleware only                 │
│  Port: 8000 (hardcoded)                         │
│  Includes: api/router.py (as api_v1_router),    │
│            atlassian_router, state_matrix_router │
│  Role: Main AI + chat + economy + pet + ...     │
│  REPL mode: threaded uvicorn on 0.0.0.0:8000    │
└─────────────────────────────────────────────────┘
```

### 1.1 Which app does the desktop connect to?

The desktop (`api-client.js`) needs `/api/v1/economy/balance/{user_id}`, `/api/v1/pet/action/trigger`, `/api/v1/angela/chat`, `/api/v1/dialogue` — all of which exist **only in App B**. Therefore the desktop **must** connect to App B.

The mobile app (`App.js`, `client.js`) calls `/api/v1/mobile/status`, `/api/v1/state/summary` — these exist in **both** apps (state_matrix in App B only, mobile.py in both).

## 6. Known Architectural Issues

### ~~6.1 Semantic Route Mismatch~~ ✅ Fixed

8 routes in `main_api_server.py:848-855` previously all mapped to `get_brain_dividend()`. Now each has its own handler:
- `GET /api/v1/desktop/state` → `DesktopInteraction.get_desktop_state()`
- `POST /api/v1/desktop/organize` → `DesktopInteraction.organize_desktop()`
- `POST /api/v1/desktop/cleanup` → `DesktopInteraction.cleanup_desktop()`
- `GET /api/v1/actions/status` → `ActionExecutor.get_execution_stats()`
- `POST /api/v1/actions/execute` → `ActionExecutor.handle_autonomous_action()`
- `POST /api/v1/tactile/touch` → `TactileService.simulate_touch()`
- `POST /api/v1/brain/metrics` → `DigitalLifeIntegrator.get_formula_metrics()` (returns ALL formula metrics)
- `POST /api/v1/brain/dividend` → unchanged (returns CDM dividend data)

### 6.2 Two Apps, Same Port

App A (`main.py`) and App B (`main_api_server.py`) both default to port 8000. They cannot run simultaneously. The desktop requires App B-specific routes; the mobile can work with either.

### ~~6.3 EncryptedCommunicationMiddleware Asymmetry~~ ✅ Fixed

Both App A and App B now have `EncryptedCommunicationMiddleware` with identical KeyB configuration.
The middleware verifies HMAC-SHA256 signatures for these path prefixes:
`/api/v1/mobile/*`, `/api/v1/system/status/detailed`, `/api/v1/system/module-control`.

### 6.4 Config Sections Never Read

~11 config sections (300+ lines in angela_core.yaml) have no Python code consuming them. These represent stale documentation or planned-but-unimplemented features.

### 6.5 `mobile.py` Duplicate GET/POST `/status`

Both `GET /mobile/status` (line 27) and `POST /mobile/status` (line 59) have IDENTICAL implementation code (return CPU/mem/cluster status). The POST version accepts a body but ignores it. Harmless but redundant.

### 6.6 CLI `core_services.py` Stub

The CLI (`packages/cli/cli/main.py`) now imports from `core_services.py` — a stub created at `apps/backend/src/core_services.py` that returns empty `_Services` objects. The real services are in `apps/backend/src/services/`. This means CLI callbacks that call `get_services().get("chat_service")` always get `None` and show "not available" messages.
